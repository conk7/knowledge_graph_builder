%     Концепция RAG (Retrieval-Augmented Generation): Кратко объясни, как RAG решает проблему галлюцинаций LLM, предоставляя контекст из базы пользователя.

%     Лексический поиск (BM25) и его лимиты: Опиши поиск по ключевым словам. Он отлично ищет специфические термины, но падает, если пользователь использует синонимы.

%     Семантический поиск (Векторные эмбеддинги) и его лимиты: Объясни принцип косинусного сходства (Cosine Similarity). Он находит контекстуально похожие тексты, но может упустить точные формулировки или выдать "популярный" мусор из-за большой разницы в векторном пространстве.

%     Гибридный поиск: Как объединение лексического и семантического поиска решает часть проблем, но всё еще буксует на сложных многошаговых запросах (multi-hop reasoning).

%     Зачем RAG-системам нужен Граф Знаний (GraphRAG): Опиши state-of-the-art подход (с опорой на статью Edge et al., 2024). Поиск, идущий по ребрам графа, позволяет LLM собирать информацию из разных, семантически далеких, но логически связанных заметок.

%     Языковые модели в задаче связывания: Краткий обзор моделей, которые ты будешь использовать. Чем тяжелые LLM (Gemini/Llama) отличаются от легковесных моделей естественного логического вывода (NLI / Cross-encoders, например, DeBERTa) в контексте задачи классификации связей.

% 1.3. Критический анализ существующих решений для построения графов знаний

% Это классический "Обзор аналогов". Твоя задача — показать, что идеального инструмента для твоей целевой аудитории пока нет.

%     Группа A: Корпоративные графовые базы данных (Enterprise Graph DBs)

%         Примеры: Neo4j Graph Builder, Memgraph.

%         Как работают: Строят двухуровневые графы (лексический и сущностный), используют кластеризацию.

%         Недостатки для PKM: Требуют развертывания инфраструктуры (дорого и сложно), не умеют инкрементально обновляться при изменении одной заметки (надо пересчитывать всё), страдают от "data hairballs" (запутанных клубков данных) из-за слепого доверия LLM без предварительной фильтрации.

%     Группа B: Академические пайплайны

%         Примеры: EduKG Approach (Ain et al., 2025).

%         Как работают: Экономят ресурсы за счет маленьких NLP-моделей (SqueezeBERT) и бьют граф на "микро-графы".

%         Недостатки для PKM: Жесткая привязка к существующим онтологиям (сложно добавить новые, неакадемические термины), требуют участия человека на каждом этапе.

%     Группа C: Облачные приложения и локальные плагины

%         Примеры: Mem.ai (облако), Smart Connections, Khoj, Graph Analysis (плагины Obsidian).

%         Недостатки для PKM: Облачные — это "черный ящик" с риском потери данных (vendor lock-in) и нарушением NDA. Плагины — либо решают проблему "в лоб" (только косинусное сходство без сложной логики), либо страдают от проблемы "холодного старта" (ориентируются на метаданные, а не на контент текста).

%     Вывод по разделу: Формулирование ниши для твоего алгоритма. Нужен локальный/гибридный, вычислительно эффективный пайплайн, который не просто ищет связи, но типизирует их, отсеивает мусор (через NLI) и аппаратно сохраняет обратно в файлы (write-back).
