\subsection{Технологии поиска и обработки естественного языка в системах RAG}

Использование генеративных моделей в чистом виде
влечет высокие риски «конфабуляций» и «галлюцинаций»:
ответ модели может казаться правдоподобным, но содержать фактологические
или причинно-следственные ошибки.

Для решения этой проблемы была разработана технология
Retrieval-Augmented Generation (RAG) \cite{Lewis_2020}.
Она добавляет дополнительный этап поиска информации перед генерацией ответа.
На основе запроса пользователя алгоритм извлекает наиболее релевантную
информацию из доступных источников, таких как текстовые файлы или базы данных,
после чего передает ее языковой модели в качестве контекста.
Такой подход заставляет LLM выступать в роли анализатора,
а не источника фактов.

Эффективность RAG-систем зависит от качества алгоритмов извлечения.
Всвязи с этим, существует несколько подходов к поиску необходимых данных.
Глобально их можно разделить на лексический и семантический поиск.

Лексический поиск основан на точных совпадениях
лексем из запроса пользователя с содержащимися в корпусах.
Данный подход отлично справляется с поиском специфических терминов,
имет собственных и прочих точных формулировок, но
не справляется в случае замены слов на синоными, чередующихся корней,
форма слов, отсутствующих в оригинальных текстах.

Для преодоления этих ограничений используется семантический поиск,
основанный на векторном сходстве.
Исходный текст разбивается на фрагменты (чанки), для каждого из которых вычисляется
многомерный числовой вектор (эмбеддинг) из одного пространства.
Поиск происходит путем вычисления сходства между этими векторами
с помощью метрик, таких как косинусное сходство.
Если полученное значение больше некоторого порога, то
чанки считаются релевантными.
Этот подход позволяет улавливать смысл текста,
не основываясь на конкретных словах или формулироваках.

Векторый поиск также имеет несколько недостатков \cite{Gao_2023}:
\begin{itemize}
      \item Потеря конкретики: смысловое значение фрагментов усредняется,
            что ведет к потере деталей, таких как имен собственных, использованных терминов, и так далее.
            Чем больше чанк, тем больше деталей теряется.
      \item Проблема «дальних эмбеддингов»: схожие данные могут иметь сильно
            отличные друг от друга векторы, из-за чего могут не быть найдены.
\end{itemize}

Современные RAG системы комбинируют эти подходы в
так называемый гибридный поиск. Он позволяет
компенсировать недостатки одного подхода с помощью другого.

Несмотря на все преимущества такого подхода,
обработка сложных аналитических запросов,
требующих многошагового рассуждения,
требует более сложных подходов.
В случаях, когда нужная информация напрямую не связана с запросом,
она не может быть найдена
ни семантическим, ни лексическим поиском, из-за чего
поисковый алгоритм RAG не сможет извлечь необходимые данные.

Более продвинутой стратегией поиска явлется Graph RAG \cite{Edge_2024}.
В ней в алгоритм поиска дополняется обходом графа знаний.
Для первоначального сбора вершин используется гибридный поиск.
Затем начинается итеративный обход соседей на основе
имеющихся ребер и их типов, что позволяет системе собирать дополнительный
контекст из файлов, которые могут не содержать ключевых слов, или быть семантически далекими,
но при этом логически связанными и обладающими необходимой информацией,
что позволяет значительно расширить контекст.

\textbf{Языковые модели в задаче связывания.}
Поскольку необходимо искать тексты, обладающие не только высоким семантическим сходством
или содержащие ключевые слова, нужно использовать иные от сравнения эмбеддингов подходы для анализа
связности текстов. Для решения этой задачи используются модели
машинного обучения, такие как Cross Encoder и LLM.
Они позволяют проверять тексты на наличие причинно-следственных, логических
и иных сложных связей.

Cross Encoder являются сравнительно небольшими моделями для попарной проверки текстов.
Эти модели принимают на вход два текста и совместно их обрабатывают с помощью cross-attention,
что позволяет выявить более тонкие логические связи и точнее провести отфильтрацию.

Большие языковые модели (LLM)
отлично понимают естественный язык и способны выявлять нетривиальные, многошаговые
логические цепочки, которые могли быть упущены при фильтрации через Cross Encoder.
Кроме того, LLM могут комбинировать фильтрацию с типизацией отношений,
что позволяет решить обе задачи за один запрос.
Однако такой подход является вычислительно сложным и, как следствие, дорогим, из-за чего
LLM используют только для классификации и дополнительной фильтрации уже отфильтрованных пар.


% ### 1.2. Технологии поиска и обработки естественного языка в системах RAG

% Интеграция больших языковых моделей (LLM) в
% системы управления персональными знаниями кардинально изменила подходы к взаимодействию
% с информацией. Однако использование генеративных моделей в чистом виде сопряжено
% с критическим недостатком — склонностью к конфабуляциям, или «галлюцинациям».
% Модель может генерировать правдоподобные, но фактически неверные утверждения,
% поскольку её ответы опираются исключительно на параметрическую память,
% сформированную на этапе предварительного обучения.

% Для решения этой проблемы была разработана архитектура
% **RAG (Retrieval-Augmented Generation)** (Lewis et al., 2020).
% Суть технологии заключается во внедрении этапа информационного поиска
% (retrieval), предшествующего генерации ответа. В ответ на запрос
% пользователя система сначала извлекает наиболее релевантные фрагменты из
% персональной базы знаний, а затем передает их языковой модели в качестве
% жесткого контекста. Таким образом, LLM выступает не как источник фактов,
% а как аналитический движок, синтезирующий ответ строго на основе
% предоставленных пользовательских данных.

% Эффективность RAG-систем напрямую зависит от качества модуля извлечения
% информации. Исторически алгоритмы поиска прошли несколько этапов эволюции,
% каждый из которых имеет свои ограничения при работе с персональными базами знаний.

% #### Лексический и семантический поиск: ограничения классических подходов
% Базовым методом извлечения в системах управления знаниями является
% **лексический поиск**, основанный на ключевых словах (в частности,
% семейство алгоритмов TF-IDF и BM25). Этот метод отлично справляется
% с поиском специфической терминологии, имен собственных и точных формулировок.
% Тем не менее алгоритм BM25 обладает высокой чувствительностью к лексическому
% составу запроса: он терпит неудачу, если пользователь использует синонимы,
% аббревиатуры или перефразирования, отсутствующие в оригинальном тексте заметки.

% На смену лексическому поиску пришел **семантический (векторный) поиск**.
% Текстовые данные с помощью моделей-энкодеров преобразуются в многомерные
% числовые векторы (эмбеддинги) и помещаются в единое векторное пространство.
% Релевантность определяется математически — путем вычисления косинусного сходства
% (Cosine Similarity) между вектором запроса и векторами документов.
% Это позволяет алгоритму улавливать общий смысл текста, игнорируя разницу в конкретных словах.

% Однако векторный поиск также имеет фундаментальные недостатки (Gao et al., 2023):
% 1. **Потеря специфичности:** Усреднение семантики документа в единый
% вектор может привести к потере важных, но редко встречающихся терминов.
% 2. **Проблема «дальних эмбеддингов»:** Контекстуально важные данные
% могут не попасть в выборку, если их эмбеддинги оказались слишком далеки
% от вектора запроса в многомерном пространстве.

% Для нивелирования недостатков обоих методов современные поисковые движки
% используют **гибридный поиск**, комбинирующий точные лексические совпадения
% (BM25) и семантическое сходство векторов. Несмотря на значительное
% улучшение метрик извлечения, гибридный подход продолжает испытывать
% трудности при обработке сложных аналитических запросов, требующих
% **многошагового логического вывода (multi-hop reasoning)**.
% Если для ответа на вопрос требуется синтезировать информацию
% из трех разных заметок, которые напрямую не связаны ни лексически,
% ни семантически, гибридный поиск с высокой долей вероятности упустит
% ключевые звенья этой цепи.

% #### Интеграция графов знаний (Graph RAG)
% Наиболее передовым подходом, преодолевающим ограничения гибридного
% поиска в задачах со сложной логикой, является **Graph RAG** (Edge et al., 2024).
% Данная парадигма расширяет стандартный пайплайн за счет использования графа
% знаний в качестве навигационной структуры.

% Вместо того чтобы искать изолированные фрагменты текста по всей
% базе, алгоритмы графового извлечения начинают поиск с узлов, наиболее
% релевантных первоначальному запросу, а затем итеративно «путешествуют»
% по типизированным семантическим ребрам. Это позволяет системе собирать
% релевантный контекст из семантически далеких, но логически связанных заметок.
% Графовый подход обеспечивает LLM упорядоченным, иерархически структурированным
% контекстом, радикально повышая полноту и точность генерации при ответах на комплексные вопросы.

% #### Языковые модели в задаче связывания
% Для автоматизации построения таких графов необходимы механизмы
% интеллектуального анализа текста. В рамках разработки пайплайна
% автоматического связывания заметок предполагается использование
% каскадной архитектуры языковых моделей, разделяющей вычислительную
% нагрузку между легковесными и тяжеловесными решениями:

% 1. **Малые модели логического вывода
% (NLI / Cross-encoders):**
% На первом этапе целесообразно использование
% компактных энкодеров (например, на архитектуре DeBERTa).
% Данные модели специализируются на задачах Natural Language
% Inference (распознавание текстового следования) и
% кросс-внимании. Их задача — быстро, вычислительно эффективно и
% детерминированно отфильтровать потенциальные пары заметок на предмет
% наличия логической связи, отсекая очевидный шум и ложноположительные результаты.
% 2. **Большие языковые модели (LLM):** Тяжелые генеративные модели
% обладают глубоким пониманием контекста и мощными способностями к
% логическому рассуждению. В пайплайне построения графа они применяются
% исключительно на финальном этапе: для анализа отфильтрованных,
% высоковероятных пар заметок и точной классификации типа семантической
% связи между ними (типизации ребра). Использование LLM только для типизации
% (а не для полнотекстового поиска) позволяет значительно снизить время
% обработки графа и затраты на использование моделей.
